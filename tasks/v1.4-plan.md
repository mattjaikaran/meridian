# Meridian v1.4 — Detailed Implementation Plan

## Phase 15: `/meridian:learn` — Execution Learning System
**Priority: HIGH | Effort: MEDIUM**

Capture execution patterns and mistakes as persistent rules injected into future subagent prompts.

### New files:
- `scripts/learnings.py` (~250 lines)
- `skills/learn/SKILL.md` (~70 lines)
- `tests/test_learnings.py` (~300 lines)

### Modify:
- `scripts/db.py` — Schema v5 migration: add `learning` table + `review.model` column
- `scripts/state.py` — Learning CRUD wrappers
- `skills/execute/SKILL.md` — Query learnings, inject into subagent prompt
- `prompts/implementer.md` — Add `## Learnings` section placeholder
- `SKILL.md` — Register command

### Schema (v5):
```sql
CREATE TABLE IF NOT EXISTS learning (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id TEXT DEFAULT 'default' REFERENCES project(id),
    scope TEXT DEFAULT 'project' CHECK (scope IN ('global', 'project', 'phase')),
    phase_id INTEGER REFERENCES phase(id),
    rule TEXT NOT NULL,
    source TEXT CHECK (source IN ('manual', 'execution', 'review', 'debug')),
    created_at TEXT DEFAULT (datetime('now')),
    applied_count INTEGER DEFAULT 0
);
ALTER TABLE review ADD COLUMN model TEXT DEFAULT 'claude';
```

### Functions (`scripts/learnings.py`):
```python
def add_learning(conn, rule, scope='project', source='manual', phase_id=None) -> dict
def get_learnings(conn, project_id='default', scope=None, limit=20) -> list[dict]
def get_learnings_for_prompt(conn, project_id='default', phase_id=None) -> str
def increment_applied(conn, learning_id) -> None
def deduplicate_check(conn, new_rule, project_id='default') -> dict | None
def prune_stale(conn, min_applied=0, older_than_days=90) -> int
def delete_learning(conn, learning_id) -> bool
```

### Skill commands:
```
/meridian:learn "always run migrations before seeding"     → Manual capture
/meridian:learn --list                                     → Show all learnings
/meridian:learn --prune                                    → Remove stale rules
/meridian:learn --delete <id>                              → Remove specific learning
```

### Auto-capture triggers:
- After plan failure → fix → success: prompt "Save this pattern as a learning?"
- After review REQUEST_CHANGES: extract the correction as a learning
- After `/meridian:debug` resolution: offer to save root cause pattern

### Deduplication:
Simple word-overlap similarity (Jaccard index on tokenized words). If >70% overlap
with existing learning, show the existing one and ask user to update or skip.
No external deps — pure Python set operations.

### Tests (~30):
- CRUD operations (add, get, list, delete)
- Scope filtering (global, project, phase)
- Deduplication detection (similar, identical, different)
- Prompt generation format
- Applied count incrementing
- Pruning (stale removal, threshold checks)
- Edge cases (empty DB, no learnings, max limit)

### Implementation:
- Uses existing `settings` table: `key='freeze_directory'`, `value='/abs/path'`

- Displayed in `/meridian:status` as: `Edit lock: src/auth/ (active)` or `Edit lock: none`

## Phase 18: Office Hours Mode for `/meridian:plan`
**Priority: MEDIUM | Effort: LOW**

Add `--deep` flag that forces 5 strategic questions before brainstorming.

### New files:
- `tests/test_office_hours.py` (~100 lines)

### Modify:
- `skills/plan/SKILL.md` — Insert pre-brainstorm question step when `--deep` is present

### The 5 forcing questions:
1. **Who needs this?** — Specific user/persona, not "everyone"
2. **What's the status quo?** — How is this solved today?
3. **What's the narrowest wedge?** — Smallest version that delivers value
4. **What breaks without it?** — Consequences of not building this
5. **What does success look like?** — Concrete, measurable outcome

### Implementation:
- No Python module needed — purely a SKILL.md procedure change
- When `--deep` is present, Claude asks all 5 questions via AskUserQuestion
- Answers stored as decisions (category: `constraint`) linked to the phase
- Answers fed into context gathering as structured context
- Shapes acceptance criteria: "success looks like" answer becomes a criterion

### Tests:
- Validate question set completeness
- Decision storage with constraint category
- Integration with plan context (answers appear in context doc)

**Priority: LOW | Effort: LOW**

### Functions (`scripts/sessions.py`):
```python
def register_session(meridian_dir: str) -> str          # Returns PID file path
def unregister_session(meridian_dir: str) -> bool
def count_active_sessions(meridian_dir: str) -> int     # Prunes stale automatically
def cleanup_stale(meridian_dir: str) -> int              # Returns count removed
```

### Implementation:
- PID files stored in `.meridian/sessions/<pid>` (content: timestamp)
- On any `/meridian:*` command invocation, register current PID
- `count_active_sessions()` checks each PID file, removes dead processes
- Status output: `Active sessions: 3`
- Execute subagent prompt addition when 3+:
  "Multiple concurrent sessions detected. Re-read shared files before modifying."

- Count accuracy with mixed live/dead

## Migration Strategy

### Schema v5 (single migration, applied in Phase 15):
```python
def _migrate_v5(conn):
    conn.execute("""
        CREATE TABLE IF NOT EXISTS learning (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            project_id TEXT DEFAULT 'default' REFERENCES project(id),
            scope TEXT DEFAULT 'project' CHECK (scope IN ('global', 'project', 'phase')),
            phase_id INTEGER REFERENCES phase(id),
            rule TEXT NOT NULL,
            source TEXT CHECK (source IN ('manual', 'execution', 'review', 'debug')),
            created_at TEXT DEFAULT (datetime('now')),
            applied_count INTEGER DEFAULT 0
        )
    """)
    # Add model column to review table (safe — column may not exist)
    try:
        conn.execute("ALTER TABLE review ADD COLUMN model TEXT DEFAULT 'claude'")
    except sqlite3.OperationalError:
        pass  # Column already exists
```

### Execution order:
1. Phase 15 (learn) — includes v5 migration
2. Phase 16 (freeze) — no migration, ships immediately after
3. Phases 17-19 — parallel, no dependencies on each other
4. Phase 20 (cross-model) — uses model column from v5 migration
